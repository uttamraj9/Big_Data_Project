# Bronze Layer — Data Ingestion (PostgreSQL → HDFS → Hive)

## Overview

This directory implements the **Bronze Layer** of the data pipeline — the raw data landing zone. It ingests synthetic credit card fraud transaction data from a PostgreSQL source database into HDFS and makes it queryable via a Hive external table.

> **Training Environment Notice**
> This is a simulation, not a production pipeline. PostgreSQL acts as a stand-in for a real OLTP source system. The raw dataset is a synthetic CSV file that is pre-split into three subsets to simulate the three ingestion patterns (full load, daily incremental, streaming) that would exist in a live environment.

---

## Architecture

```
[Synthetic CSV]
      |
      | simulate_data_split.py  (one-time setup)
      |
      v
[data/split/]
  full_load.csv          (60% — initial historical batch)
  incremental_load.csv   (20% — simulated daily new records)
  kafka_streaming.csv    (20% — simulated Kafka stream)
      |
      | ingest_to_postgres.py  (full / inc / stream)
      |
      v
[PostgreSQL — testdb]
  cc_fraud_trans           (main transaction table)
  cc_fraud_streaming_data  (streaming simulation table)
      |
      | Sqoop (MapReduce)
      |
      v
[HDFS Bronze Zone]
  /tmp/US_UK_05052025/class_project/input/raw_data_sqoop/
      |
      | Hive External Table (DDL only, no data movement)
      |
      v
[Hive — bd_class_project.cc_fraud_trans]
```

## Dataset

| Field | Description |
|---|---|
| `Transaction_ID` | Unique transaction identifier |
| `User_ID` | User account identifier |
| `Transaction_Amount` | Transaction value in currency |
| `Transaction_Type` | POS / ATM Withdrawal / Bank Transfer / Online |
| `Timestamp` | Transaction date and time |
| `Account_Balance` | Account balance at time of transaction |

| `Merchant_Category` | Restaurants / Travel / Groceries / Electronics / Clothing |

| `Failed_Transaction_Count_7d` | Failed transactions in the past 7 days |

| `Transaction_Distance` | Distance (km) from user's typical location |

## How to Run

### Step 1 — Prepare simulation data (one-time)
Splits the raw CSV into the three simulation subsets:
```bash
cd data_ingestion_postgres
python3 src/simulate_data_split.py
```

### Step 2 — Load initial data into PostgreSQL (full load)
```bash
cd data_ingestion_postgres/src
ENV_FILE=../.env python3 ingest_to_postgres.py full
```

### Step 3 — Full load to HDFS and create Hive table (one-time)
Run from the Cloudera edge node (where Sqoop and beeline are available):
```bash
export DB_HOST=<postgres-ip> DB_NAME=testdb DB_USERNAME=admin DB_PASSWORD=admin123
bash bronze_full_load.sh
```

### Step 4 — Daily incremental load (or run manually)
```bash
# First simulate new records arriving in PostgreSQL
ENV_FILE=../.env python3 ingest_to_postgres.py inc

# Then push them to HDFS
bash bronze_incremental_load.sh
```

## Jenkins Automation

The `Jenkinsfile` automates the daily incremental cycle at **10:00 AM**:

1. **Install Dependencies** — creates/reuses a Python virtualenv
2. **Simulate New Records in PostgreSQL** — runs `ingest_to_postgres.py inc`
3. **Incremental Load: PostgreSQL → HDFS Bronze** — runs `bronze_incremental_load.sh`

Jenkins reads all DB credentials from the `my-env-file` file credential (configured in Jenkins Credentials Manager).

## Notes

- The `data/` directory is excluded from git (`.gitignore`) as it contains large CSV files.
- The `.env` file is excluded from git — never commit credentials.
- The Hive table is **EXTERNAL** — dropping it in Hive does not delete the HDFS files.
- Sqoop uses `--m 1` (single mapper) since this is a small training dataset.
- The incremental load uses a **timestamp watermark** pattern: it queries `MAX(Timestamp)` from Hive and imports only rows newer than that value.
